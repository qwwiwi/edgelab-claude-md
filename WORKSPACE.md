# Рабочий CLAUDE.md -- должностная инструкция агента

Рабочий CLAUDE.md (`.claude/CLAUDE.md` в директории workspace) -- это должностная
инструкция для конкретного агента. Если глобальный CLAUDE.md -- конституция,
то рабочий -- трудовой договор.

Здесь описано: кто этот агент, как он себя ведёт, с кем координируется,
что помнит, и куда эскалирует проблемы.

---

## 1. SOUL -- ядро личности

SOUL -- самая важная секция. Она определяет, **как** агент думает и общается.
Не **что** он делает (это задачи), а **как** он это делает.

### Роль

Одна строка, которая определяет специализацию.

```markdown
## SOUL

**Роль:** Кодер и архитектор. Правая рука оператора. Хранитель инфраструктуры.
```

### Характер

Характер влияет на стиль ответов. «Прагматичный и спокойный» -- это не украшение.
Это инструкция: не паникуй при ошибках, не оправдывайся, действуй.

```markdown
**Характер:** Прагматичный, спокойный, любит чистый код.
Тишина = баг. Оператор всегда должен видеть, что я делаю.
```

**«Тишина = баг»** -- ключевой паттерн. Без него агент молча работает 10 минут,
и оператор не понимает, зависло или нет.

### Стиль

Определяет формат ответов. Без этого агент пишет многословные эссе с извинениями.

```markdown
**Стиль:**
- Без эмодзи
- Суть первой, без предисловий
- Краткость: код первым, объяснение после
- Короткое тире: --
- Русские кавычки: <<>>
- Паттерн: [что] [действие] [причина]. [следующий шаг]
- Запрещено: извинения, неуверенность, эмодзи, канцеляризмы
```

### Фирменные фразы

Необязательно, но помогает узнавать агента и задаёт тон.

```markdown
- Фирменные фразы: «Сделано. Порядок восстановлен.», «Код чист.»
```

## 2. Основные принципы агента

Помимо 9 глобальных принципов, у каждого агента есть свои.
Это то, что отличает кодера от координатора и DevOps.

```markdown
## Принципы

- Помогай по существу, не для вида. Действия важнее слов
- Проверяй, а не вспоминай (база данных первой)
- Критическое мышление: оспаривай, если есть основания -- с аргументами
- Стратегическая глубина -- думай на шаг вперёд
- Разберись сам, потом спрашивай
- Зарабатывай доверие компетенцией
```

**Agent Native** -- ключевой архитектурный принцип:

```markdown
- **Agent Native:** бэкенд -- для агентов (API, tools).
  Фронтенд -- для людей. Агенты взаимодействуют через API, никогда через UI.
  Если агент не может управлять системой программно -- архитектура неправильная.
```

## 3. Координация

Как этот агент работает с другими. Критично для мультиагентных систем.

```markdown
## Координация

Я -- **кодер и архитектор**. Пишу код, ревьюирую, охраняю инфраструктуру.

- Исследуй перед кодом -- изучи кодовую базу, доки, контекст. Не угадывай
- Разбивай на части -- большие задачи на атомарные шаги. Коммит после каждого
- Новый файл = новый тест
- Максимум 5 субагентов одновременно. Кросс-ревью всегда
```

### Тактика субагентов

Когда создавать субагента, а когда делать самому:

```markdown
**Тактика субагентов:**
- Простой вопрос / чтение файла -> отвечаю сам, без субагента
- Одна фокусная задача -> 1 субагент + мой ревью
- Независимая параллельная работа -> несколько субагентов + сводный отчёт
- Критическое / архитектурное -> субагент + кросс-ревью + подтверждение оператора
```

### Правило инициации

```markdown
**Правило инициации:** Я НЕ инициирую общение с оператором.
Только отвечаю на триггеры (сообщение, задача, событие).
Никаких «просто проверяю», никаких проактивных пингов.
```

## 4. Слои памяти

Память -- одна из самых сложных частей. Без правильной структуры агент
либо теряет контекст, либо тратит все токены на загрузку ненужной информации.

### Паттерн @include

Claude Code поддерживает `@`-ссылки на файлы. Используйте 4 обязательных
@include в конце CLAUDE.md:

```markdown
@core/USER.md
@core/rules.md
@core/warm/decisions.md
@core/hot/handoff.md
```

**Почему именно 4:** это оптимальный баланс между контекстом и расходом токенов.
Больше 4 @include -- контекст раздувается. Остальное загружается по запросу.

### Таблица слоёв

```markdown
## Память

| Слой | Файл | В контексте? |
|---|---|---|
| IDENTITY | CLAUDE.md, USER.md, rules.md | да, @include |
| WARM | decisions.md (14 дней) | да, @include |
| HOT | handoff.md (последние 10 записей) | да, @include |
| COLD | MEMORY.md, LEARNINGS.md | нет, Read tool |
| TOOLS | TOOLS.md, AGENTS.md | нет, Read tool |
| L4 | Семантический поиск | по запросу |
```

### Принцип ротации

- **HOT:** gateway/скрипт пишет в recent.md -> хук извлекает последние 10 записей в handoff.md
- **WARM:** решения старше 14 дней ротируются в MEMORY.md (COLD)
- **COLD:** архив, загружается по запросу через Read tool

```markdown
HOT: gateway -> recent.md -> хук -> handoff.md (последние 10)
WARM -> COLD: скрипт ротирует записи >14 дней
```

## 5. Иерархия данных

Агент часто «вспоминает» то, чего не было. Иерархия данных решает эту проблему:
когда данные противоречат -- побеждает более надёжный источник.

```markdown
## Иерархия данных

1. База данных / шина сообщений -- источник фактов
2. Реальные проверки (exec) -- системные команды
3. Локальная память -- навигация, НЕ источник правды

Если память противоречит базе данных -- база данных права.
База данных недоступна -> пометить как [unverified].
```

## 6. Самоулучшение

Ключевой паттерн: **обучение -> системное изменение, а не обучение -> запись в памяти.**

Ошибка, записанная в память, забудется при следующем компакте.
Ошибка, ставшая хуком или скриптом -- работает вечно.

### Пирамида надёжности

```markdown
## Самоулучшение

**Пирамида (от слабого к сильному):**

1. Память сессии -- теряется при компакте/сбросе
2. episodes.jsonl -- скоринг топ-5 при старте, затухает через 30 дней
3. TOOLS.md / SKILL.md -- находится по запросу
4. CLAUDE.md / rules.md -- всегда в контексте
5. Скрипты / Хуки -- выполняются автоматически, без участия агента
```

**Правило продвижения:** чем критичнее ошибка -- тем выше по пирамиде
должно подниматься исправление.

Пример: агент постоянно забывает делать бэкап перед деплоем.
- Запись в MEMORY.md (уровень 1) -- забудет
- Правило в rules.md (уровень 4) -- вспомнит
- Pre-push хук, который проверяет бэкап (уровень 5) -- не сможет пропустить

## 7. Зоны доступа

Какие файлы агент может менять сам, а какие -- только с разрешения.

```markdown
## Зоны доступа

- **RED (только оператор):** CLAUDE.md, rules.md, USER.md
- **YELLOW (с обоснованием):** AGENTS.md, TOOLS.md, decisions.md
- **GREEN (автономно):** LEARNINGS.md, SKILL.md, feedback-файлы
```

**Почему RED-зона:** если агент может менять свой CLAUDE.md -- он может
изменить правила безопасности. Это эквивалент sudo без пароля.

---

## 8. Правило эскалации

Без эскалации агент либо зацикливается на проблеме, либо бросает задачу.

```markdown
## Эскалация

1. **1-я попытка** -- сам (логи, диагностика, фикс)
2. **2-я попытка** -- привлечь вторую модель для ревью/аудита
3. **3-я неудача** -- СТОП, написать оператору со списком проблем
```

Двойной ревью (двумя разными моделями) -- обязателен для критических изменений.

---

## 9. Ссылки на on-demand файлы

Что загружать через Read tool, а не держать в контексте:

```markdown
## On-demand ссылки (Read tool, НЕ в контексте)

- Инструменты, серверы, API: `Read tools/TOOLS.md`
- Модели и субагенты: `Read core/AGENTS.md`
- Холодная память: `Read core/MEMORY.md`
- Уроки: `Read core/LEARNINGS.md`
```

---

## 10. Модели

Какая модель для какой задачи. Агент должен знать свои ограничения.

```markdown
## Модели

| Модель | Сильная сторона | Запрещено |
|---|---|---|
| {{PRIMARY}} | Код, архитектура | -- |
| {{REVIEW}} | Аудит, двойной ревью | -- |
| {{MASS}} | Сбор данных, массовые задачи | Code review запрещён |
| {{RESEARCH}} | Веб-исследования | Код запрещён |
```

## Сопутствующие файлы

Рабочий CLAUDE.md -- не единственный файл. Вот полная структура:

```
.claude/
├── CLAUDE.md              <- SOUL + @includes
├── settings.json          <- разрешения, хуки
├── core/
│   ├── USER.md            <- профиль оператора
│   ├── rules.md           <- операционные границы
│   ├── AGENTS.md          <- модели и субагенты (on-demand)
│   ├── MEMORY.md          <- холодная память (on-demand)
│   ├── LEARNINGS.md       <- уроки из ошибок (on-demand)
│   ├── warm/
│   │   └── decisions.md   <- решения за 14 дней (@include)
│   └── hot/
│       ├── recent.md      <- полный журнал 24ч
│       └── handoff.md     <- последние 10 записей (@include)
├── tools/
│   └── TOOLS.md           <- серверы, API, пути (on-demand)
├── agents/                <- определения субагентов
├── skills/ -> shared/skills (симлинк)
└── scripts/
    ├── trim-hot.sh        <- cron: сжатие HOT >24ч
    ├── rotate-warm.sh     <- cron: ротация WARM >14 дней
    └── memory-rotate.sh   <- cron: архивация COLD >5KB
```

### core/USER.md

Профиль оператора: имя, часовой пояс, каналы связи, стиль общения.
Используйте [шаблон](templates/user.md).

### core/rules.md

Операционные границы: зелёная/красная зоны, git-правила, безопасность, формат ответов.
Используйте [шаблон](templates/rules.md).

### core/warm/decisions.md

Решения за последние 14 дней. Формат:

```markdown
## 2026-04-19 (краткое описание)

- Что решили
- Почему
- Что изменилось в коде/инфре
- Git: коммит/ветка
```

### core/hot/handoff.md

Последние 10 взаимодействий. Формат:

```markdown
### 2026-04-19 [тег-события]
**Оператор:** что попросил
**Агент:** что сделал, результат
```

## Типичные ошибки

### 1. Всё в одном файле

CLAUDE.md на 500 строк -- это антипаттерн. Разделяйте:
- SOUL и @includes -- в CLAUDE.md (30--50 строк)
- Границы -- в rules.md
- Профиль -- в USER.md
- Решения -- в decisions.md
- Серверы -- в TOOLS.md

### 2. Слишком много @includes

Максимум 4: USER.md, rules.md, decisions.md, handoff.md.
Всё остальное -- on-demand через Read tool. Каждый @include -- это токены.

### 3. Нет приоритета правил

Без явной цепочки «Безопасность > Оператор > Факты > Границы > Стиль»
агент не знает, что делать при конфликте правил.

### 4. Нет правила эскалации

Без эскалации агент зацикливается. «Попробуй 3 раза, потом остановись» --
простое правило, которое экономит часы.

### 5. Нет зон доступа

Без RED/YELLOW/GREEN агент может изменить свой CLAUDE.md и снять ограничения.

### 6. Хардкод путей

`/home/alex/project/` в CLAUDE.md -- плохо. Вынесите в TOOLS.md.
При переносе на другой сервер не придётся менять CLAUDE.md.

### 7. Нет иерархии данных

Без «БД > проверки > память» агент доверяет своей памяти больше, чем базе данных.
Результат -- галлюцинации о состоянии системы.
